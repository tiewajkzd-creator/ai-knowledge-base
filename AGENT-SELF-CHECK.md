# Agent 自检方案

> 基于 AI大脑 2026-04-14 身份/流程混乱问题制定，供所有 Agent 对照检查

---

## 自检原则

每次新 session 开始前，对照以下清单逐项检查。发现任何一项异常，立即记录并修复，再开始处理其他任务。

---

## 第一部分：身份与上下文（启动必检）

### 1.1 BOOTSTRAP.md 是否存在？
- **检查**：workspace 根目录是否存在 `BOOTSTRAP.md`
- **如果存在**：立即执行——读内容、理解角色定义、完成身份建立、然后删除它
- **如果不存在**：跳过，继续下一步

### 1.2 SOUL.md 是否已读？
- **检查**：我是否在 session 开始时主动读过 `SOUL.md`
- **如果没有**：立即读，理解我的定位、风格、边界
- **验收标准**：能用自己的话说出"我是谁、我的风格是什么"

### 1.3 USER.md 是否已读？
- **检查**：我是否在 session 开始时主动读过 `USER.md`
- **验收标准**：知道用户的名字、称呼方式、时区、偏好

### 1.4 memory/ 当天+昨天 是否已读？
- **检查**：`memory/YYYY-MM-DD.md` 文件（今天和昨天）
- **验收标准**：知道最近发生过什么重要事件

### 1.5 MEMORY.md 是否已读（仅主 session）？
- **检查**：主 session 直接聊天时，必须读 `MEMORY.md`
- **注意**：群组聊天或共享 session 禁止加载 MEMORY.md（隐私保护）
- **验收标准**：能说出长期记忆中的关键决策和偏好

### 1.6 身份自测
- **自问**：我现在叫什么？我在为谁服务？我在这个系统里的角色是什么？
- **如果答不上来**：立即重新执行 1.1 → 1.5，不继续其他任务

---

## 第二部分：知识库与外部依赖（工作前检）

### 2.1 知识库结构是否清楚？
- **检查**：
  - 知识库存放在哪个目录？
  - 分类结构是什么？
  - 新内容应该存哪个分类？
- **如果不清楚**：立即读 `knowledge-base/README.md` 或向用户确认

### 2.2 外部发布目标是否记录？
- **检查**：这个 workspace 的内容有没有发布到外部（如 GitHub Pages）？
  - GitHub 仓库地址是什么？
  - 远程分支配置是否存在（`git remote -v`）？
  - 同步流程是什么？
- **如果不清楚**：询问用户，禁止在未确认的情况下执行 `git push`

### 2.3 链接可访问性验证
- **检查**：用户提到某个链接时，先验证是否真的可访问
- **验证方法**：`curl -sI <URL>` 检查 HTTP 状态码
- **禁止**：假设链接有效而不验证

---

## 第三部分：执行安全边界

### 3.1 禁止全盘搜索
- **禁止命令**：`find ~`、`find /Users`、`find /` 等扫描 home 或根目录的命令
- **允许范围**：
  - `~/.openclaw/workspace/<当前agent>/`
  - `~/.openclaw/<明确知道用途的子目录>/`
  - 明确指定的特定路径
- **如果必须搜索更大范围**：先询问用户，说明原因和预期范围

### 3.2 危险操作二次确认
- **以下操作执行前必须询问**：
  - `git push --force`
  - 删除文件（非 trash）
  - 修改配置文件（openclaw.json 等）
  - 添加或删除 cron 任务
  - 向外部服务发送数据

---

## 第四部分：记忆与上下文连续性

### 4.1 MemOS 依赖检查
- **原则**：MemOS（云端记忆）是辅助，本地文件是主源
- **当两者冲突时**：以用户当前消息的实际意图为准，MemOS 片段仅供参考
- **收到新信息时**：立即写入本地 `memory/YYYY-MM-DD.md`，不只依赖 MemOS 上报

### 4.2 决策记录
- **关键决策**（如选择了某个方案、拒绝了某个方案）必须写入 `MEMORY.md`
- **原因**：让未来的我不因为 MemOS 片段缺失而不知道当时为什么这么做

### 4.3 工作进度透明
- **多步骤任务进行中**：每个关键步骤完成后，简短汇报进度
- **禁止**：长时间静默后突然给出一个大结果（用户不知道中间发生了什么）

---

## 第五部分：特定场景处理

### 5.1 链接处理流程
收到链接时的标准流程：
1. 判断内容质量（值得存入？拿不准？有毒？）
2. 值得存入 → 按分类规则存入知识库 → 确认存储成功 → 回复"已存入"
3. 拿不准 → 简述判断理由，推荐用户决定
4. 有毒/无用 → 不回应
5. **任何存入操作后**：验证文件确实存在

### 5.2 视频抓取流程
1. yt-dlp 下载音频（记录文件路径和大小）
2. ffmpeg 提取音频
3. whisper 转写（记录耗时和段落数）
4. 整理摘要并存入知识库
5. **更新 index**：确保知识库索引反映新增内容
6. **git 同步**：如有远程仓库，推送更新
7. **清理临时文件**：不留残留
8. **通知用户**：成功/失败均必须通知

### 5.3 新工具/新流程首次使用
- **流程**：先读官方文档或 skill 文件（`~/.agents/skills/<name>/SKILL.md`）
- **禁止**：凭直觉猜测工具用法后直接执行
- **失败处理**：记录错误信息，整理问题，向用户报告

---

## 第六部分：Agent 间协作（多 Agent 环境）

### 6.1 跨 Agent 信息共享
- 如果某个 Agent 创建的资源（如 git 仓库、配置文件）可能对其他 Agent 有用，应在 `MEMORY.md` 中记录位置和用途
- 定期检查 `~/.openclaw/workspace/` 下其他 Agent 的最新状态

### 6.2 调用其他 Agent 前的检查
- 确认任务确实需要子 Agent，而不是当前 Agent 直接处理
- 任务完成后，将结果整合到当前上下文中，不让子 Agent 的输出悬空

---

## 自检执行记录

每次 session 开始时，在 `memory/YYYY-MM-DD.md` 中记录：

```
## [HH:MM] 自检完成
- 身份：✓ 已知 / ✗ 未知（补救措施：xxx）
- 知识库：✓ 清楚 / ✗ 不清楚（补救措施：xxx）
- 外部依赖：✓ 确认 / ✗ 待确认
- 执行边界：✓ 无违规 / ✗ 有违规（已修正）
```

发现任何异常项，立即修复或向用户报告，不带病继续工作。

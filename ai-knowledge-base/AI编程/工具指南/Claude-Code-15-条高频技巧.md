## 技巧 1：设置 Claude Code 别名

传统启动命令很长：
```bash
claude -p --model sonnet
```
可以设置别名（如 `c-d`），一键启动最高权限实例，大幅提升启动效率。

---

## 技巧 2：用 /mini 重构或初始化 CLAUDE.md

CLAUDE.md 是 Claude Code 的**记忆文件**，每次对话内容都会累积加入。

**不会写？用 /mini：**
- 让 Claude Code 自己了解代码库，生成初始 CLAUDE.md
- 建议按以下结构调整：

| 区块 | 内容 |
|------|------|
| 项目简短描述 | 1-2句话 |
| 代码规范 | 引用外部文件（不要堆太多） |
| 常用 npm | 启动/测试命令 |
| 主要文件结构 | 目录树 |
| AI 必须记住的事 | 删掉后代码质量会下降的信息 |

> **判断标准：** 删掉这条信息后，大模型返回的代码质量会变差/爆错 → 留下；否则删掉。

**新版本 /mini 更智能：** 会搜索代码库信息 + 配置技能/插件/钩子。
- 在 `settings.json` 开启 `claude.code.newEditor: 1`，使用新版流程。

## 技巧 3：不喜欢终端？用其他 UI

- **VS Code 官方插件**：内置 Claude Code 能力
- **Web Dashboard**：网页版 UI，可视化管理多任务/多 worktree
  - 支持看板视图（任务状态：进行中/已完成/review）
  - 支持多开 worktree 实例

## 技巧 4：Esc 键的妙用

| 操作 | 效果 |
|------|------|
| 按一次 Esc | 停止输入 |
| 连按两次 Esc | 直接进入 /rewind 模式回滚 |
| 关闭实例后 | 出现恢复命令，可用对应 mini 重启 |

- 可给重启命令设置快捷键
- 也可用 `c-d rewind` 的别名方式重启

## 技巧 5：提供详细的反馈链路

**❌ 没有反馈链路的做法：**
> "帮我做一个邮箱密码更换功能" → Claude 写完代码 → "完成了"

**✅ 有反馈链路的做法：**
> 任务 → 编译 → 代码测试 → 页面测试 → 流程测试 → 全部通过 = 真正完成

**好处：** 极大提高准确率。链路中哪步失败，Claude 会继续修复——不是简单写完代码就算成功。

> 很多 SPIKE 规范驱动的工具会自动生成对应场景的测试用例，涵盖这个链路。

## 技巧 6：安装语言 LSP 插件

LSP（Language Server Protocol）= 每次文件编辑完自动纠错（类型错误、未使用变量、语法错误等）。

**没有 LSP →** 问题要到编译阶段才发现
**有 LSP →** Claude Code 可以**提前发现**问题

**安装步骤：**
1. 在 VS Code 搜索并安装官方推荐的语言插件
2. 在 Discover 搜索 LSP → 选中安装 → 选择用户级/工程级

## 技巧 7：尽可能多使用技能 / 创造技能

**技能 = 把一套流程封装好 + 让大模型给出确定结果**

技能的价值：**减少 AI 返回代码的不确定性。**

适用环节：
- 头脑风暴 → 需求拆解 → UI设计 → 技术设计 → /task 开发 → 测试 → 上线 → 复盘

**每个节点都有好用的技能可用。**

- 通过 claude-code-skills 网站搜索需要的技能
- 使用 `claude skill add` 命令安装（自动适配不同 AI 编程工具的技能目录）

## 技巧 8：不确定时用 Plan 模式（／plan）

**Plan 模式 = Claude Code 内置的子 Agent：**
- 搜索代码库所有相关文字
- 跟你沟通需求 → 确定需求文档 → 再执行代码

**好处：** 你和 AI 之间的需求逻辑完全对齐，不会出现理解偏差。

**切换方式：** 连续按 `Shift + Tab` 看到 Plan 模式按钮，即可开始对话。

---

## 技巧 9：不要转述 Bug，直接贴报错

**❌ 转述（有风险）：**
> "登录跳转有问题，可能是 token 没存，帮我修一下"

Claude 会直接按你的猜测去排查，如果你的怀疑是错的，就错过了真正的修复机会。

**✅ 贴原始信息（推荐）：**
> "登录跳转有问题" + **附上原始报错日志**
> + 可加上自己的怀疑方向

Claude 会参考你的意见，同时分析原始日志排查所有可能性。

## 技巧 10：无关任务用 /clear 清上下文

**需要清上下文的情况：**
1. 新任务与上一任务**完全无关**，不需要历史上下文
2. AI 一直修不好、绕来绕去 → clear 一下重新来，Claude 会"完全清醒"

> 上下文不断累积会触发压缩，压缩会损失对话质量，影响 AI 输出结果。

---

## 技巧 11：用子代理保持主上下文清晰

**子代理 = 独立上下文，不占用主对话的上下文**，相当于开了个后台线程。

**适合用子代理的场景：**

| 类型 | 说明 |
|------|------|
| **Code Review** | 主代理要的是 review 结果，不需要关注过程 |
| **专家型任务** | 支付漏洞扫描等专项任务 |
| **大量输出任务** | 运行测试、生成文档等会产生大量文字的 |
| **并行分发** | 拆给多个子任务同时执行（需保持子任务相互独立） |

> 注意：同一代码库上多子代理可能相互交叉影响，需设计好隔离策略。

---

## 技巧 12：用 /btw 快速提问（不占上下文）

`/btw` = 在 Claude Code 运行过程中，额外发起对话，**不污染主上下文**。

**场景：** 任务运行中，突然想问一个相关问题 → 用 /btw 开小窗 → 对话记录不占主上下文。

---

## 技巧 13：用 --worktree 并行开发

```bash
claude --worktree 分支名
```
- 每个 worktree 是一个独立分支 + 独立实例
- 多个 worktree 并行开发，互不影响
- 可搭配 Web Dashboard 界面化管理

> 适合能同时管住多个任务的人。如果多任务容易混乱，可以先从双线开始。

---

## 技巧 14：选择合适的 MCP

MCP（Model Context Protocol）可以直连外部系统：
- **Playwright MCP**：自动打开浏览器做 UI 测试
- **MySQL MCP**：直连数据库，测试数据是否插入成功
- 比执行脚本的方式效率高很多

---

## 技巧 15：告诉 Claude Code 要看哪些文件

虽然 Claude Code 搜索代码能力很强，但**你更懂代码结构**：

- 用 `/apatrol` 告诉它哪些文件相关
- 直接把认为有问题的文件拖给它
- 让它精准定位，而不是让它全库搜索

> 这样做也能**节省 token 消耗**。

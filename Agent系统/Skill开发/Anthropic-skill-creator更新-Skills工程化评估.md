## 核心观点

### 1. Skill 评估困境
过去 Skills 最大问题不是不会写，而是**写完后无法严肃评估**：
- 不知道是否比裸模型更强
- 模型升级后是否还有价值
- 只能靠感觉判断好坏

### 2. 工程化转变
Anthropic 更新 skill-creator，首次把"做 skill"从手工作坊推向工程化流程，提供一套方法判断 **skill 是否值得存在**。

### 3. 完整闭环
- **Eval**：设计测试样本验证任务完成度
- **Benchmark**：记录通过率、耗时、token 成本
- **多代理并行测试**：独立上下文避免串场
- **Description 优化**：自动调整描述减少误触发

背后是多类子代理协同：执行、打分、盲评对比、整体模式分析。

### 4. 官方案例
- **PDF skill**：评估后发现在非可填写表单里容易失败，定位到坐标锚点不稳定，修复后真正解决
- **Description 优化**：Anthropic 测试自己公开的 6 个 skills，有 5 个触发准确率都提升

### 5. 两类 Skill

**能力增强型**：
- 提升输出质量（拆结构化笔记、转 PPT、大幅提升某类输出）
- 核心问题：今天是否明显强于裸模型？
- 如果差距不大，可能该退休了

**工作流约束型**：
- 固定顺序/标准/格式输出（会议纪要、周报、公众号改写）
- 不一定比裸模型"更聪明"，但能让模型按你的规则来
- 价值在于**稳定**，而非炫技

### 6. 实践建议

**更新流程**：
1. 更新 skill-creator 到最新版（`https://github.com/anthropics/skills/tree/main/skills/skill-creator`）
2. 挑 1-2 个最常用的旧 skills
3. 跑一遍 eval 和 benchmark
4. 看通过率、耗时、token 和对比结果
5. 决定保留/优化 description/重写

**核心转变**：
- 过去：凭感觉判断 skill 好坏
- 现在：用数据证明 skill 价值
- 未来：skill 像软件一样有版本管理、性能监控、持续优化

## 我的消化

### 核心突破
**可测试性**是 AI Agent 工程化的关键痛点。这次更新让 Skills 从"手工作坊"进入"工程化流程"。

### 对 AI大脑+知识库 系统的启发

**1. 知识库内容评估**
- 借鉴 eval 思路，设计测试样本验证采集质量
- 例：随机抽取 10 条内容，检查是否符合过滤标准

**2. 采集管道优化**
- benchmark 通过率/耗时/成本，找出瓶颈
- 记录每个平台的抓取成功率和平均耗时

**3. 白名单动态调整**
- 定期评估博主内容质量（命中率、互动率、相关度）
- 自动淘汰低价值源，发现高质量新源

**4. 工作流约束型思维**
- 采集流程不需要"更聪明"，需要"更稳定"
- 统一小结模板就是一种工作流约束
- 去重引擎、过滤规则都是约束型组件

### 可迁移原则

1. **任何自动化流程都需要可测试性**
   - 不能只靠"感觉还行"
   - 要有明确的成功/失败标准

2. **稳定性比偶尔惊艳更重要**
   - 90% 稳定输出 > 10% 惊艳 + 90% 不确定

3. **工具要当成长期资产管理**
   - 不是一次性脚本
   - 需要版本管理、性能监控、持续优化

## 📌 核心定位

ComfyUI 全新内存优化系统，徹底改变扩散模型的显存管理方式，让显存受限的设备也能流畅运行超大型开源模型。

---

## 🎯 六大核心改进

| 改进项 | 说明 |
|--------|------|
| **系统内存占用更低** | 运行复杂工作流所需的常规内存显著减少 |
| **彻底消除 OOM 错误** | 因权重卸载不足导致的内存溢出崩溃问题完全解决 |
| **加载速度更快** | 部分场景下模型初始加载与 LoRA 应用速度大幅提升 |
| **避免系统分页** | 无需依赖操作系统缓慢的分页文件，即可运行超出物理内存容量的模型 |
| **显存利用率更高** | GPU 显存占用可能比以往更高，属于正常现象（系统正在更高效利用最快显存） |
| **简化开发流程** | 不再需要预估模型内存占用并预留空间 |

## ⚙️ 技术原理：AI 模型动态卸载器（aimdo）

动态显存是专为 PyTorch 定制的显存分配器，在原生 PyTorch 分配器压力过大时按需卸载模型权重。

### 1. 虚拟基地址寄存器（VBAR）
- 创建 VBAR **完全不占用物理显存**，仅消耗 GPU 虚拟地址空间（基本无成本）
- 在 VBAR 内为模型权重分配张量，初始状态下**完全未分配物理空间**

### 2. fault() API（即时分配）
- 程序不一次性加载全部内容，而是在计算需要张量的瞬间**按需调入**
- **此时才真正占用物理显存**

### 3. 分配成功 vs 分配失败

| 情况 | 行为 |
|------|------|
| **显存充足** | 为权重分配显存，数据写入后保留在显存以保证速度；出现显存压力时可立即释放 |
| **显存不足** | ComfyUI **不会 OOM 崩溃**，而是创建临时 GPU 张量仅复制当前层所需数据，执行完毕后释放或复用 |

### 4. 优先级与内存水位（watermark）机制
- 最近加载的 VBAR（当前活跃模型）拥有**最高优先级**
- 高优先级权重需要空间时会**强制驱逐低优先级权重**
- 被驱逐的权重所在位置会设置内存水位，同一 VBAR 内位于水位之上的权重调用 `fault()` 时将自动失败
- 这一机制使应用能够**高效判断显存空间**，避免反复尝试加载造成资源浪费

## 🔑 全新内存行为

### safetensors 加载器
- 采用更高效的文件打开方式，**避免提交内存分配**
- 文件以映射至**未提交文件内存**的方式打开
- 权重通过指针直接指向未提交内存，而非深度拷贝至 PyTorch 模型
- 模型加载节点几乎**瞬间执行**

### 与旧系统的本质区别
- **不再将显存中的模型卸载回内存**（卸载回内存是 OOM 的根源）
- 未提交内存分配伴随模型整个生命周期（跨流程运行也保留）
- 减少 PCIe 与内存总线传输，避免多模型工作流中卸载模型导致的内存耗尽
- 模型释放显存后可瞬间恢复到未提交加载状态

### Windows 任务管理器说明
- Windows 用户常看到内存占用较高——系统会尽可能缓存数据，但需要时可直接回收
- **缓存权重绝不会被写入分页文件**
- Linux 系统监视器中，未提交内存显示为"缓存"而非"已用"，因此内存占用看起来很低
